# OpenClaw 多智能体协作部署指南

> 适用版本：v2026.1.6+
> 前提条件：已完成 OpenClaw 单智能体的基础部署

---

## 一、理解多智能体是什么

在 OpenClaw 中，**一个智能体（Agent）** 就是一个独立的 AI 助手实例，拥有自己的：

- **工作区**：存放文件、配置、个人规则（SOUL.md / AGENTS.md）
- **状态目录**：认证信息、模型配置
- **会话记录**：对话历史互不干扰

**多智能体协作** 的核心价值：

- 不同的智能体负责不同的频道/用户/任务
- 各智能体可配置不同的模型、工具权限、人格设定
- 智能体之间可以按需互相调用（需显式开启）

**形象比喻**：把每个智能体想象成一个独立员工，他们共用同一台服务器（Gateway），但各自有独立的办公室（工作区）、不同的专长（工具配置）、互不干扰的工作记录（会话）。

---

## 二、整体架构

```
OpenClaw Gateway（单一入口）
        │
        ├── 路由规则（Bindings）
        │
        ├── Agent: main        ← 默认智能体
        │     ├── 工作区: ~/.openclaw/workspace
        │     └── 会话: ~/.openclaw/agents/main/sessions
        │
        ├── Agent: coder       ← 编程助手
        │     ├── 工作区: ~/.openclaw/workspace-coder
        │     └── 会话: ~/.openclaw/agents/coder/sessions
        │
        └── Agent: analyst     ← 数据分析助手
              ├── 工作区: ~/.openclaw/workspace-analyst
              └── 会话: ~/.openclaw/agents/analyst/sessions
```

---

## 三、路由规则（消息如何分配给智能体）

OpenClaw 按优先级匹配规则，**越具体的规则优先级越高**：

| 优先级 | 规则类型         | 说明                |
|-----|--------------|-------------------|
| 1   | 精确匹配对等点      | 指定具体的 DM/群组/频道 ID |
| 2   | 父对等点继承       | 子频道继承父频道规则        |
| 3   | Discord 角色路由 | 按服务器角色分配          |
| 4   | Guild/服务器级别  | 整个 Discord 服务器    |
| 5   | 团队级别（Slack）  | 整个 Slack 工作区      |
| 6   | 账户 ID 匹配     | 某个频道账户的所有消息       |
| 7   | 频道通配符        | 整个频道平台            |
| 8   | 默认智能体        | 兜底匹配              |

> 同一优先级下，**配置文件中排在前面的规则先生效**。

## 四、部署步骤

### 4.1 创建新智能体

```bash
# 创建一个名为 coder 的智能体
openclaw agents add coder

# 创建一个名为 analyst 的智能体
openclaw agents add analyst
```

执行后，OpenClaw 会自动创建：

- `~/.openclaw/workspace-coder/`
- `~/.openclaw/agents/coder/`

此时查看openclaw.json会发现多了两个agent的配置

### 4.2 查看已有智能体

```bash
openclaw agents list
```

### 4.3 修改配置文件

配置文件路径：`~/.openclaw/openclaw.json`

在现有配置基础上，添加多智能体的 `list` 和 `bindings`（绑定/路由规则,作用是：决定收到的消息交给哪个智能体处理） 配置：

```json
{
  "agents": {
    "defaults": {
      "model": {
        "primary": "vllm/Qwen3.5-27B"
      },
      "workspace": "/home/lane/.openclaw/workspace",
      "compaction": {
        "mode": "safeguard"
      }
    },
    "list": [
      {
        "id": "main",
        "name": "通用助手",
        "model": {
          "primary": "vllm/Qwen3.5-27B"
        }
      },
      {
        "id": "coder",
        "name": "编程助手",
        "model": {
          "primary": "vllm/Qwen3.5-27B"
        },
        "tools": {
          "profile": "coding"
        }
      },
      {
        "id": "analyst",
        "name": "数据分析助手",
        "model": {
          "primary": "vllm/Qwen3.5-27B"
        },
        "tools": {
          "profile": "minimal"
        }
      }
    ],
    "bindings": [
      {
        "comment": "来自 Telegram 账户 personal 的消息 → 通用助手",
        "agentId": "main",
        "match": {
          "channel": "telegram",
          "accountId": "personal"
        }
      },
      {
        "comment": "来自 Telegram 账户 work 的消息 → 编程助手",
        "agentId": "coder",
        "match": {
          "channel": "telegram",
          "accountId": "work"
        }
      }
    ]
  }
}
```

其中第一个 bindings 的意思如下：凡是在 Telegram 中发消息给 personal 这个bot的消息，都交给
main（通用助手）处理，此时的channel配置得有以下这样的配置：

```json
  {
  "channels": {
    "telegram": {
      "accounts": {
        "personal": {
          "botToken": "111111:AAAAxxxxxx"
        },
        ...
      }
    }
  }
}
```

以上是不同的bot使用不同的agent，无关具体用户，但是也可以针对具体用户使用什么agent进行配置，详细的看后面。

### 4.4 验证配置

```bash
# 查看所有智能体及其绑定关系
openclaw agents list --bindings
```

### 4.5 重启服务

```bash
# 重启 openclaw 服务使配置生效
sudo systemctl restart openclaw
# 或者
openclaw restart
```

## 五、常见场景配置示例

### 场景一：同一个 Telegram，不同用户找不同智能体

```json
"bindings": [
{
"comment": "张三的消息 → 编程助手",
"agentId": "coder",
"match": {
"channel": "telegram",
"peer": {"kind": "direct", "id": "123456789"}
}
},
{
"comment": "李四的消息 → 数据助手",
"agentId": "analyst",
"match": {
"channel": "telegram",
"peer": {"kind": "direct", "id": "987654321"}
}
},
{
"comment": "其余所有 Telegram 消息 → 通用助手（兜底）",
"agentId": "main",
"match": {
"channel": "telegram"
}
}
]
```

其中

```text
"agentId": "coder",
"match": {
"channel": "telegram",
"peer": {"kind": "direct", "id": "123456789"}
}
```

这条表示，只要是Telegram 用户id是123456789的，不管和哪个bot聊天，都走coder这个agent。而以下：

```json
"agentId": "main",
"match": {
"channel": "telegram",
"accountId": "personal",
"peer": "123456789"
}
```

表示只要是Telegram 用户id是123456789的用户使用personal这个bot聊天，都使用main这个agent

> Telegram 用户 ID 可以在 BotFather 的消息日志或第三方工具中获取。



---

### 场景二：群组中通过 @ 触发不同智能体

```json
"list": [
{
"id": "coder",
"name": "编程助手",
"mentionPatterns": ["@coder", "@编程助手"]
},
{
"id": "analyst",
"name": "数据助手",
"mentionPatterns": ["@analyst", "@数据助手"]
}
]
```

在群组中，用户输入 `@coder 帮我写一段Python代码` 就会触发编程助手。

## 六、智能体间互相调用（Agent-to-Agent）

默认情况下智能体之间**不能互相通信**，需要在配置中显式开启：

```json
"list": [
{
"id": "main",
"tools": {
"agentToAgent": {
"enabled": true,
"allow": ["coder", "analyst"]
}
}
},
{
"id": "coder",
"tools": {
"agentToAgent": {
"enabled": true,
"allow": ["main"]
}
}
}
]
```

**白名单说明**：`allow` 列表中的是**可以被当前智能体调用的**目标智能体 ID。

## 七、为每个智能体设置独立人格

在各自的工作区目录下创建 `SOUL.md` 文件：

```bash
# 为编程助手设置人格
cat > ~/.openclaw/workspace-coder/SOUL.md << 'EOF'
你是一位专业的全栈工程师，擅长 Python、JavaScript 和系统架构设计。
回答问题时请简洁精准，优先给出可运行的代码示例。
EOF

# 为数据助手设置人格
cat > ~/.openclaw/workspace-analyst/SOUL.md << 'EOF'
你是一位数据分析专家，擅长 SQL、数据可视化和统计分析。
回答时请结合数据思维，多提供分析视角和结论。
EOF
```

## 八、为每个智能体配置工具权限（v2026.1.6+）

不同智能体可以开放或限制不同的工具：

```json
"list": [
{
"id": "coder",
"tools": {
"profile": "coding",
"allow": ["bash", "read_file", "write_file", "search"
],
"deny": [
"camera", "contacts"]
},
"sandbox": {
"mode": "all",
"scope": "agent",
"docker": {
"setupCommand": "pip install numpy pandas"
}
}
},
{
"id": "analyst",
"tools": {
"profile": "minimal",
"allow": ["read_file", "search"]
}
}
]
```

**sandbox 说明**：

- `mode: "all"` — 所有工具调用都在沙箱（Docker 容器）内执行
- `scope: "agent"` — 每个智能体独立沙箱，不共享
- `setupCommand` — 容器首次创建时自动执行的初始化命令

## 十、常见问题

### Q: 新增智能体后消息还是被旧智能体处理？

**A**: 检查 `bindings` 的顺序，更具体的规则必须排在前面。另外确保执行了 `openclaw restart` 重新加载配置。

### Q: 如何查看某条消息被哪个智能体处理了？

**A**: 查看日志：

```bash
journalctl -u openclaw -f
# 或
openclaw logs --follow
```

### Q: 多个智能体会互相影响会话上下文吗？

**A**: 不会。每个智能体的会话完全隔离，存储在 `~/.openclaw/agents/<agentId>/sessions/` 下，互不干扰。

### Q: 多智能体会增加太多资源消耗吗？

**A**: OpenClaw 只有在智能体**实际处理消息时**才会调用模型，空闲的智能体不占用计算资源。内存开销主要来自会话缓存，可通过
`compaction` 设置控制。

### Q: 智能体的工作区文件在哪里？

**A**: 默认路径规则：

- main 智能体：`~/.openclaw/workspace/`
- 其他智能体：`~/.openclaw/workspace-<agentId>/`
