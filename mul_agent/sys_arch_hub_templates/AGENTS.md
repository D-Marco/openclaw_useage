# 多智能体协作规范（sys_arch 中转模式）

本工作区用于飞书多智能体协作。

本模式的唯一协调中枢是 `sys_arch`。`be_coder` 与 `fe_coder` 负责各自专业工作，但所有阶段性结果、阻塞、交付物路径、完成通知，都必须先回传给 `sys_arch`，再由 `sys_arch` 统一对外通知飞书群聊并决定下一步分发。

## 启动规则

每次会话启动后，先读：

1. `SOUL.md`
2. `AGENTS.md`
3. 当前任务相关文档

不要先提问再行动。优先阅读上下文、识别当前阶段、继续推进任务。

## 总体原则

- 所有产出必须落盘，禁止只在聊天里给结果不写文件。
- 文档优先写入 `output/docs/`
- 源码优先写入 `output/src/`
- SQL 优先写入 `output/sql/`
- 每次新增或修改关键文件后，都要在消息中明确告知文件路径。
- agent 间传递内容时，优先传“文件路径 + 简短说明”，不要粘贴整篇文档或大段源码。

## 中转机制

这是本模式最重要的规则：

- `be_coder` 不直接向 `fe_coder` 发送阶段完成通知。
- `fe_coder` 不直接向 `be_coder` 发送阶段完成通知。
- 所有阶段结果统一回传给 `sys_arch`。
- 由 `sys_arch` 负责：
  - 汇总进度
  - 通知飞书群聊
  - 将后端产物转交前端
  - 将前端问题转交后端
  - 在最终阶段对外汇报整体完成情况

## sessionKey 规则

这是避免 announce 丢失的关键：

- 回传 `sys_arch` 时，必须使用 `sys_arch` 在本轮任务中明确提供的 `callback_session_key`
- 禁止把阶段完成消息发送到 `agent:sys_arch:main`
- 禁止假设 `main` 会自动回到飞书群
- 如果你收到的是来自 `sys_arch` 的任务消息，优先从任务正文读取 `callback_session_key`
- 如果任务正文没写，但系统上下文中出现了 `Agent 1 (requester) session: ...`，且请求方是 `sys_arch`，则回传该精确 session key

推荐任务消息格式：

```text
任务目标：
...

输入文件：
- ...

完成后回传：
- callback_session_key: agent:sys_arch:feishu:group:oc_xxx
- 回传内容必须包含：产出文件路径、完成摘要、阻塞项（如有）
```

## sessions_send 规则

- 使用 `sessions_send` 进行 agent 间协作
- 默认只传路径、摘要、待办，不传大块正文
- 阶段完成时，向 `sys_arch` 回传如下信息：
  - 完成了什么
  - 产出文件路径
  - 是否可进入下一阶段
  - 是否需要其他角色协助
- 如遇阻塞，也先回传 `sys_arch`，由 `sys_arch` 统一协调

推荐回传格式：

```text
阶段状态：已完成
工作摘要：已完成用户认证接口与接口文档整理
产出文件：
- output/docs/api-auth.md
- output/src/auth/...
下一步建议：可开始登录页与鉴权流程联调
阻塞项：无
```

## 飞书通知规则

- 只有绑定在飞书群上的会话，才负责对飞书群聊发消息或发附件
- `sys_arch` 是默认对外发言人与附件发送者
- `be_coder` 与 `fe_coder` 产出文件后，只需回传路径给 `sys_arch`
- 若需要将文件发送到飞书群，由 `sys_arch` 使用绑定会话统一发送

## sys_arch 分发规则

当 `sys_arch` 向其他 agent 派发任务时，必须同时提供：

- 任务目标
- 输入文档路径
- 交付物要求
- 回传时机
- `callback_session_key`

若需要前后端串联，规则如下：

1. `sys_arch` 先产出需求与架构文档
2. `sys_arch` 派发后端任务给 `be_coder`
3. `be_coder` 完成接口或后端阶段成果后，回传给 `sys_arch`
4. `sys_arch` 读取后端产物，补充前端所需说明，再派发给 `fe_coder`
5. `fe_coder` 完成后回传给 `sys_arch`
6. `sys_arch` 统一向飞书群汇报

## 文档和源码质量要求

- 文档要结构化，避免只有结论没有依据
- 源码要与文档保持一致
- 接口文档必须写明请求、响应、鉴权、错误码、字段含义
- 前端文档必须写明页面、状态流、接口依赖、异常态
- 后端文档必须写明模块边界、表结构、接口边界、关键流程

## 阻塞处理

遇到以下情况，不要沉默：

- 输入文档缺失
- 依赖接口定义不完整
- 表结构与需求冲突
- 前后端约定不一致
- 当前阶段无法继续推进

此时立即回传 `sys_arch`，说明：

- 阻塞原因
- 受影响文件
- 需要谁决策
- 建议方案

## 禁止事项

- 禁止把阶段完成直接通知给另一个 coder，绕开 `sys_arch`
- 禁止把结果默认回发到 `agent:<id>:main` 期待自动 announce
- 禁止仅在聊天中给出长篇结果而不落盘
- 禁止将敏感信息、密钥、密码写入文档
- 禁止未经确认执行破坏性命令

## 成功判定

本模式下，一个阶段真正“完成”的标准不是“自己做完了”，而是同时满足：

1. 产出已落盘
2. 路径已明确
3. 已按要求回传 `sys_arch`
4. `sys_arch` 已能基于该结果继续编排下一阶段
