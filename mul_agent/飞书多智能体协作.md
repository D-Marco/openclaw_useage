# OpenClaw 飞书多智能体协作

> 适用版本：v2026.1.6+

OpenClaw 支持两种多 agent 飞书群聊玩法，请根据实际需求选择对应文档：

---

## 玩法一：Binding 模式

**文档**：[飞书多智能体-binding模式.md](./飞书多智能体-binding模式.md)

**特点**：不同飞书群绑定不同 agent，每个群只有一个 agent 响应。

```
系统设计群  →  sys_arch
后端开发群  →  be_coder
前端开发群  →  fe_coder
```

**适合场景**：按团队/职能划分群组，各群有明确的职责边界。

---

## 玩法二：Broadcast 模式

**文档**：[飞书多智能体-broadcast模式.md](./飞书多智能体-broadcast模式.md)

**特点**：一个飞书群，消息广播给所有 agent。飞书渠道中 sys_arch 始终是唯一回复 Feishu 的 agent（由 bindings 决定），be_coder / fe_coder 作为观察者静默记录上下文，由 sys_arch 通过内部调用驱动。

```
同一个群  →  @ 机器人  →  sys_arch 回复飞书
                        sys_arch 按需内部调用 be_coder / fe_coder
```

> **注意**：飞书 broadcast 模式中，`groupChat.mentionPatterns` **不生效**（源码中 Feishu 的 broadcast 分支不调用 mentionPatterns 匹配逻辑）。be_coder / fe_coder 只能由 sys_arch 通过 `sessions_send` 内部激活，不能直接从飞书触发。

**适合场景**：团队在同一个群内工作，由 sys_arch 统一接收需求并驱动后续 agent 协作。

---

## 两种模式对比

| 对比项 | Binding 模式 | Broadcast 模式 |
|-------|------------|--------------|
| 群数量 | 每个 agent 一个群 | 一个群 |
| 路由方式 | 群组 ID 决定 | bindings 决定（始终为 sys_arch） |
| 关键配置 | `bindings.peer` | `broadcast` |
| 配置复杂度 | 低 | 低 |
| 使用便利性 | 需切换群 | 在同一群内操作，但飞书侧只有 sys_arch 响应 |
