# 系统解读

## Router Agent 和 Agent-as-Tool

**Router Agent**

```sql
User
 ↓
Default Agent（路由判断）
 ↓
Ops Agent（领域处理）
```

特点：

1. 不做业务，只负责分类
2. ops agent 是完整业务 agent
3. 用户感知不到切换

**Agent-as-Tool**

```shell
User
 ↓
Primary Agent（通常默认agent）
 ↓
Ops Agent（作为一个tool被调用）

```

此时，

- ops agent 不再是独立对话主体
- 而是 default agent 的“能力插件”。

## Lobster

类似工作流
**适合：**

- 多步骤自动化
- 有审批环节
- 有副作用操作
- 需要可恢复流程

**不适合：**

- 单轮 AI 对话
- 简单工具调用
- 纯推理任务

## LLM Task

一个专门用于“结构化 LLM 任务”的插件工具。

- 仅 JSON 输出的 LLM 调用
- 可绑定 schema 校验
- 可嵌入 workflow

## Exec Tool

让 OpenClaw agent 在操作系统里执行 shell 命令

## apply_patch

代码修改工具。

原理是用 LLM 生成建议，通过 diff 来实现代码修改。

## agent send

- 用于主动调用 agent， 通过命令 openclaw agent xxx 来调用
- 定时任务执行 cli 命令，如定时生成分析报告

## sub agent

OpenClaw 的后台并行智能体执行机制
**核心特点：**

- 主 Agent 可派生子 Agent 执行任务
- 子 Agent 独立 session、独立上下文
- 非阻塞执行（后台跑）
- 完成后自动向主对话回传结果

**典型用途：**

- 长耗时 research / scraping
- 并行任务处理
- 多 agent 协同分工

## multi agent

多 agent 协同分工

- 与 sub agent 不同，multi agent 是多个 agent 并行合作完成一个任务。

- 一个 agent 可以作为 multi-agent 体系里的一个长期角色，也可以被 临时当成 sub-agent 执行一次任务，但语义完全不同。
- sub agent 角色不稳定，它的 prompt 不稳定（帮我分析这个日志：xxx.log 重点关注网络异常。），它需要拼接，而 multi agent 有固定的 prompt，虽然都将 prompt 作为入参传入，但是 multi agent 是有这样的结构 ：

  - system prompt（角色定义，长期稳定）
  - agent context（记忆、工具、策略）
  - task prompt（本次任务输入）

而 Sub-Agent 调用时：

- system prompt = 临时拼装
- context = 很少甚至没有
- task prompt = 主输入
- memory 不连续。Multi-agent 持续积累经验，sub-agent 每次都是新 agent

## Slash Commands

OpenClaw 的 Slash Commands 就是“用聊天命令控制 AI 或系统行为的一套指令”。
类似 Linux 命令行、Discord /command、或你在 AI agent 里加的控制参数，只不过它嵌在聊天里。

| 分类      | 命令          | 参数示例                  | 功能说明               | 使用场景                      |
| --------- | ------------- | ------------------------- | ---------------------- | ----------------------------- |
| 模型控制  | `/model`      | `gpt-5`, `deepseek-chat`  | 切换当前使用的模型     | 调试、成本控制、fallback 策略 |
| 推理控制  | `/think`      | `low`, `medium`, `high`   | 设置推理强度或复杂度   | 控制输出精细度、分析深度      |
| 输出调试  | `/verbose`    | `on`, `off`               | 开启/关闭详细调试信息  | 调试 agent、查看 tool 调用链  |
| 会话管理  | `/new`        | —                         | 开启新会话，清空上下文 | 避免历史对话影响              |
| 会话管理  | `/reset`      | —                         | 重置当前对话上下文     | 保留模型设置，清除历史        |
| 会话管理  | `/stop`       | —                         | 停止当前执行任务       | 中断长输出或工具执行          |
| 系统状态  | `/status`     | —                         | 查看系统和模型状态     | 监控 token 使用、模型状态     |
| 系统状态  | `/usage`      | —                         | 查看使用量统计         | 成本监控、quota 管理          |
| 系统状态  | `/help`       | —                         | 查看可用命令和用法     | 新用户或调试                  |
| 工具调用  | `/skill`      | `sql_gen`, `camera_check` | 调用特定技能或工具     | 自动化任务、Agent workflow    |
| 工具调用  | `/exec`       | `python script.py`        | 执行系统命令或脚本     | 高权限运维、自动化操作        |
| 工具调用  | `/bash`       | `ls -la`                  | 在系统 shell 执行命令  | 高权限运维、调试              |
| 配置管理  | `/config set` | `temperature=0.2`         | 修改模型或系统参数     | prompt tuning、行为调整       |
| 调试开关  | `/debug`      | `on`, `off`               | 打开/关闭调试模式      | 查看调用链、latency           |
| 身份/权限 | `/whoami`     | —                         | 查看当前用户信息       | 权限确认、多用户管理          |
| 身份/权限 | `/role`       | `admin`                   | 切换角色或权限组       | 多 Agent 系统、企业管理       |

## Skills

Skill 就是 AI 能用的“插件 / 工具能力”，可以有以下功能：

- 调 API
- 执行脚本
- 调数据库
- 自动化流程
- 浏览网页

skill 被调用的流程一般如下：

```swift
LLM（模型）
   ↓
Agent（智能体）
   ↓
Skills（能力模块）
   ↓
Tools/API/脚本
```

**Skill = Agent 调用外部能力的桥梁。**

每个 Skill 是一个目录，核心文件：

```sql
skills/
  camera-diagnose/
    SKILL.md
```

SKILL.md 文件最少内容：

```yaml
---
name: camera-diagnose
description: Diagnose camera status
---
```

### skill 的加载机制

OpenClaw 会从三个地方加载：

1. 自带 Skills（系统内置）
2. 本地 Skills（~/.openclaw/skills）
3. 工作区 Skills（workspace/skills）

优先级：工作区 Skills > 本地 Skills > 自带 Skills（即如果同名 Skill 在多个地方存在，系统按优先级选用。）

**OpenClaw 会把可用 Skills 注入 system prompt**
